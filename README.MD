# Predictive Maintenance System вЂ” Azure PdM Dataset

![Tests](https://github.com/DerKaiser7/field-service-predictive-maintenance/actions/workflows/tests.yml/badge.svg)

Binary ensemble classifier that predicts whether an industrial machine will fail within the next 24 hours, enabling proactive maintenance intervention and reducing unplanned downtime.

**[Live Demo](https://field-service-maintenance.streamlit.app) В· [Dataset](https://www.kaggle.com/datasets/arnabbiswas1/microsoft-azure-predictive-maintenance)**

> The dashboard's "Live Demo" tab runs real ensemble inference, but on 20 hand-authored machine profiles rather than live database rows вЂ” Streamlit Community Cloud has no database access. Telemetry sparklines are simulated waveforms, not sampled sensor data. See [Design Notes](#design-notes) below.

---

## Problem

Unplanned machine failures are significantly more costly than scheduled maintenance вЂ” both in repair cost and lost production time. This system uses historical sensor readings, error logs, and maintenance records to predict failures 24 hours in advance.

**ML Task:** Binary classification вЂ” predict `label=1` if a failure occurs within 24 hours of a given observation, `label=0` otherwise.

**Dataset:** Microsoft Azure Predictive Maintenance вЂ” 100 machines, 876,100 hourly telemetry observations over one year.

## Data

| Table | Rows | Description |
| --- | --- | --- |
| `machines` | 100 | Machine metadata (model, age) |
| `telemetry` | 876,100 | Hourly sensor readings (voltage, rotation, pressure, vibration) |
| `errors` | 3,919 | Error events per machine |
| `maintenance` | 3,286 | Maintenance history by component |
| `failures` | 761 | Failure events with failure type |

**Class balance:** 17,184 positive labels (1.96%) вЂ” handled via `scale_pos_weight=49.98` in XGBoost and `class_weight='balanced'` in Logistic Regression.

## Features

42 engineered features per observation, all computed strictly from data available at or before `observation_time` (no leakage):

| Category | Features |
| --- | --- |
| Static | `model`, `age`, `age_category` |
| Telemetry 3h | mean/std/min/max for voltage, rotation, pressure, vibration (16 features) |
| Telemetry 12h | mean/std for all four sensors (8 features) |
| Telemetry 24h | mean/std for all four sensors (8 features) |
| Errors | `error_count_24h`, `hours_since_last_error`, `distinct_error_types` |
| Maintenance | `days_since_last_maintenance`, `component_diversity` |
| Failure history | `total_prior_failures`, `days_since_last_failure`, `distinct_failure_types` |

**Label:** `1` if failure occurs in `(t, t+24h]`, else `0`.

See [docs/FEATURE_ENGINEERING.md](docs/FEATURE_ENGINEERING.md) for full rationale.

---

## Modelling

**Three-stage pipeline:**

1. **Baseline вЂ” Logistic Regression** (`class_weight='balanced'`, 5-fold CV): Sanity check and meta-learner input.

2. **Main вЂ” XGBoost** (`scale_pos_weight=49.98`, grid search over depth/lr/estimators): boosted model tuned on PR-AUC.

3. **Ensemble вЂ” Stacked meta-learner**: Logistic Regression trained on val-set predictions from LR + XGBoost. Threshold optimised against cost matrix: missed failure (FN) penalised 5Г— over false alarm (FP).

**Primary metric:** PR-AUC (preferred over ROC-AUC for imbalanced data).

## Results

Train/val/test split: 60/20/20 (stratified). All metrics on held-out test set.

| Model | PR-AUC | Precision | Recall | F1 |
| --- | --- | --- | --- | --- |
| Logistic Regression (baseline) | 0.8285 | 0.4735 | 0.9895 | 0.6405 |
| XGBoost | 0.9999 | 0.9840 | 1.0000 | 0.9919 |
| Ensemble (threshold=0.729) | 0.9992 | 0.9888 | 0.9985 | 0.9936 |

**Ensemble at optimal threshold (cost: 5Г—FN + 1Г—FP):**

- True Positives: 3,432 вЂ” True Negatives: 171,744
- False Positives: 39 вЂ” False Negatives: 5
- Business cost: 64

## Design Notes

Trade-offs made deliberately for a demo context:

- **Model artifacts are committed to git** (`model_artifacts/`, ~624KB total). No model registry or external storage вЂ” appropriate at this size, but would move to one (MLflow registry, S3) if artifacts grew or if multiple model versions needed tracking.
- **Live Demo profiles are hardcoded**, not queried from a database. Streamlit Community Cloud's free tier has no database access, so `dashboard/app.py` ships 20 machine profiles designed to span the LOW/MEDIUM/HIGH risk spectrum. The ensemble inference and SHAP explanations on those profiles are real; the input data is not.
- **CORS is fully open** (`allow_origins=["*"]`) in `src/api/main.py`. Demo API with no auth surface вЂ” would be scoped to specific origins before handling real traffic.
- **XGBoost's near-perfect test PR-AUC (0.9999)** was treated as a leakage red flag during development, not just a good result. `src/models/ablation_leakage_check.py` retrains the model with `total_prior_failures`, `days_since_last_failure`, and `distinct_failure_types` removed вЂ” these are the features most structurally similar to the label, since both derive from the `failures` table. Test PR-AUC drops by only 0.0003 (0.9999 в†’ 0.9995, FN: 0 в†’ 4), meaning the score is driven by telemetry/error signal, not by those features leaking future failure information. `tests/test_feature_leakage.py` adds a static regression guard against the most likely leakage mistake (a feature CTE picking up the label's forward-looking window).

## Running Locally

**Prerequisites:** Docker, Python 3.11, `.env` file with DB credentials.

```bash
# 1. Start PostgreSQL
docker compose up -d

# 2. Install dependencies
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# 3. Load data
python -m src.data_operations.load
python -m src.data_operations.promote_to_base

# 4. Build features
psql -h localhost -U $POSTGRES_USER -d $POSTGRES_DB -f sql/features/build_features.sql

# 5. Train models
python -m src.models.train_baseline
python -m src.models.train_xgboost
python -m src.models.ensemble

# 6. Run Streamlit demo (uses committed model artifacts вЂ” no DB required)
streamlit run dashboard/app.py

# 7. Run FastAPI (optional) вЂ” locally...
uvicorn src.api.main:app --port 8000

# ...or as a container alongside Postgres
docker compose up -d --build api
```
